# 认证文件按请求成功次数排序设计

## 背景

认证文件页面当前已经支持“默认”“A-Z 名称”“优先级”三种排序方式，卡片上也已经展示了每个认证文件的成功/失败请求次数。现在需要在不改动后端接口、不扩展页面范围的前提下，为“认证文件”列表新增一种按请求成功次数排序的方式，方便优先查看请求成功次数较高的认证文件。

## 目标

- 在认证文件页面的现有“排序”下拉中新增“请求成功次数”选项。
- 选择该选项后，列表按请求成功次数降序排列。
- 当两个文件的请求成功次数相同时，按修改时间倒序排列。
- 保持现有默认排序与其他排序方式不变。
- 复用现有前端统计数据来源，避免新增后端改造。

## 非目标

- 不修改默认排序规则。
- 不新增“按失败次数排序”“按总请求次数排序”等其他排序方式。
- 不新增后端接口、字段或统计口径。
- 不重构认证文件页面的筛选、分页或卡片统计实现。

## 用户界面设计

认证文件页面继续使用现有排序控件，不新增独立按钮或切换器。

在 `src/pages/AuthFilesPage.tsx` 的 `sortOptions` 中新增一项：

- `请求成功次数`

该选项加入现有下拉列表，与“默认”“A-Z 名称”“优先级”并列显示。

页面初始排序仍保持 `default`，不会自动切换到“请求成功次数”。如果页面排序状态已持久化，则新增排序值也需要被现有状态恢复逻辑接受。

## 排序行为设计

当排序模式为“请求成功次数”时：

1. 对每个认证文件读取其请求成功次数。
2. 先按成功次数从高到低排序。
3. 若成功次数相同，则按修改时间从新到旧排序。
4. 若修改时间也相同，则保持当前数组排序稳定性，不额外引入新的兜底排序规则。

该排序仅影响当前筛选结果，因此页面行为仍然是：

1. 先按状态筛选、类型筛选和搜索关键字得到结果集。
2. 再按“请求成功次数”排序。
3. 最后按现有分页逻辑切页展示。

这保证了新排序与现有搜索、筛选、分页协同工作，不改变页面其他交互顺序。

## 数据来源设计

新排序直接复用当前卡片展示成功/失败数字所使用的数据来源：

- `src/features/authFiles/constants.ts` 中的 `resolveAuthFileStats(file, keyStats)`

排序时取其返回值中的 `success` 字段作为排序依据。这样可以确保：

- 列表排序与卡片上显示的“成功 N”完全一致。
- 不需要新增额外字段或重复计算逻辑。
- 统计口径与现有页面保持统一。

当某个认证文件暂时无法匹配到统计数据时，其成功次数按 `0` 处理。这与当前卡片的默认展示行为一致。

## 状态与持久化设计

认证文件页面当前使用 `AuthFilesSortMode` 管理排序模式，并通过 `uiState` 做本地持久化。

因此需要同步扩展：

- `AUTH_FILES_SORT_MODES`
- `AuthFilesSortMode`
- `isAuthFilesSortMode()` 的合法值集合

新增的排序值建议命名为：

- `success`

这样页面切换排序后，现有的读写逻辑可以继续复用，刷新页面后也能够正确恢复该排序模式。

## 影响范围

本次设计只涉及前端管理界面，影响范围限定在以下几处：

- `src/features/authFiles/uiState.ts`
- `src/pages/AuthFilesPage.tsx`
- `src/i18n/locales/zh-CN.json`

其中：

- `uiState.ts` 负责接纳新的排序模式。
- `AuthFilesPage.tsx` 负责新增下拉选项与排序分支。
- `zh-CN.json` 负责新增“请求成功次数”文案。

如果项目要求所有语言包保持完整一致，则后续实现时可同步补齐其它 locale；如果只做最小闭环，至少需要保证中文界面正确显示该选项文本。

## 错误处理与边界

这次改动不引入新的异步请求，因此不存在额外的网络错误处理。

需要明确的边界行为只有两点：

- 统计缺失时按成功次数 `0` 处理。
- 成功次数并列时按修改时间倒序处理。

除此之外，不增加额外兜底逻辑，不改变现有列表的空态、加载态或错误态表现。

## 验证方案

实现完成后需要验证以下行为：

1. 排序下拉中出现“请求成功次数”选项。
2. 选择该选项后，成功次数更高的认证文件排在前面。
3. 当两个文件成功次数相同，修改时间更新的排在前面。
4. 切换回“默认”“A-Z 名称”“优先级”时，现有排序行为不受影响。
5. 页面刷新后，如果当前排序状态被持久化，新排序值可以被正确恢复。
6. 对没有统计数据的认证文件，页面不会报错，且其排序按成功次数 `0` 参与比较。

## 推荐实现方式

推荐直接在 `AuthFilesPage` 的现有 `sorted` 计算逻辑中新增 `success` 分支，并在分支内调用 `resolveAuthFileStats(file, keyStats).success` 进行比较。

这是本次需求改动最小的实现方式，原因是：

- 不需要新增 view model。
- 不需要改动卡片组件的数据结构。
- 不需要引入新的派生缓存。
- 能够最大程度复用现有逻辑并降低回归风险。

## 待实施摘要

最终实现应满足以下结果：

- 排序下拉新增“请求成功次数”。
- 选择后按成功次数降序排列。
- 并列时按修改时间倒序。
- 现有其余排序与页面流程保持不变。
